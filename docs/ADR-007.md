# ADR-007: Strategy Sync Engine Architecture

## Status: Accepted

## Date: 2026-06-24

## Problem
The existing backtesting system had duplicated configuration management and inline backtest logic that was difficult to maintain and extend. The BacktestContext was duplicating some config fields from the EngineConfigContext, and the SimulationTab contained ~600 lines of inline backtest logic.

## Solution
Created a pure engine architecture with:

1. **EngineConfigContext** as single source of truth for all configuration
2. **src/engine/** directory with pure functions:
   - `types.ts` — all shared interfaces
   - `ranker.ts` — ranking functions
   - `crashDetector.ts` — crash detection/recovery
   - `allocator.ts` — allocation logic
   - `metrics.ts` — metric computation
   - `core.ts` — `runStrategy()` main orchestration

3. **SimulationTab** as thin UI wrapper around the engine
4. **Custom tickers** feature for forced holdings
5. **Crash threshold slider** (5-30%) in UI
6. **Strategy Profile card** showing active profile details

## Key Benefits

### 1. Single Source of Truth
- EngineConfigContext consolidates all configuration
- Eliminated BacktestContext duplication
- Consistent configuration across all modules

### 2. Pure Functions
- All engine functions are pure (same input → same output)
- No React dependencies in engine
- Easy to test and reason about

### 3. Separation of Concerns
- UI (SimulationTab) separate from business logic (engine)
- Easy to extend with new features
- Clear boundaries between components

### 4. Extensible Architecture
- Easy to add new features (custom tickers, crash slider, etc.)
- Can be integrated with Portfolio, Market, Notification, and AI modules
- Follows DRY principle

## v2 Extension: Portfolio = Strategy Control Center (2026-06-24)

### Problem
v1 achieved single source of truth, but user wanted:
- Backtest config lebih detail (custom mode, custom universe)
- Settingan Backtest = Settingan Portfolio (sama persis setelah SYNC)
- Portfolio cascades ke Market, Notifications, AI
- AI menjawab "should I exit?" berdasarkan settingan user

### Solution

**1. Custom Mode + Custom Universe (Phase 8):**
- `simulationMode: "algo" | "single" | "custom"` (3 modes)
- `customUniverse: string[]` — separate dari `customTickers`
- 3-button mode toggle di AppSidebar
- Custom Universe picker UI (multi-select + chips)

**2. SYNC TO PORTO Real (Phase 9):**
- `syncFromBacktest(snapshot)` function di EngineConfigContext
- `StrategySnapshot` interface — full strategy config
- Replace `console.log` dengan actual sync logic
- `sonner` toast notifications
- Success toast: "Strategy synced to portfolio"

**3. Portfolio = Strategy Control Center (Phase 10):**
- Remove `activeConfig` prop dari PortfolioTracker (use context only)
- Enhanced Active Strategy banner (mode, universe, crash, safe haven, top N)
- `shouldTriggerExit()`, `evaluateStrategy()`, `getActiveUniverse()` engine helpers
- Auto-notify via NotificationContext

**4. Notification Rule Firing (Phase 11):**
- `fireRule(rule, notification)` helper
- 4 rules: `tickerOutOfTopN`, `crashProtectionTriggered`, `customUniverseBreach`, `singleModeTrigger`
- Threshold-based, not real-time

**5. Market Tab Filter Cascade (Phase 12):**
- Mode-aware filter: algo+customTickers / algo+universe / custom / single
- Memoize `visibleStocks` dengan `useMemo`
- "Filtered by Portfolio Strategy" badge

**6. AI Explain Exit Logic (Phase 13):**
- Section 12 "STRATEGY EVALUATION" di systemKnowledge
- `strategyEvaluation` + `activeUniverse` di `AILiveContext`
- `buildLiveContext()` pass new fields

### Key Design Decisions (v2)

1. **`customUniverse` (exclusive) vs `customTickers` (forced) — separate fields:**
   - Different semantics, kept separate to avoid confusion
   - `customTickers` = forced holdings in algo mode (always enter portfolio)
   - `customUniverse` = exclusive list in custom mode (replaces universe)

2. **SYNC = replace strategy fields, exclude cash:**
   - User expects "sama persis" after SYNC
   - Cash is user's money, not part of strategy

3. **Notification = threshold-based, not real-time:**
   - Avoid notification spam
   - Use `shouldFireRule` dedup mechanism

4. **Toast library = `sonner` (~3KB):**
   - Modern, lightweight, standard in React ecosystem

5. **Docs-first approach:**
   - D1-D7 documentation updates before any code changes
   - Then code (Phase 8-13)
   - Then verify (V1-V6)

### Flow

```
[Backtest] → user configures (mode/customUniverse/profile/crash) → runStrategy() → result
         ↓
[SYNC TO PORTO] → syncFromBacktest(snapshot) → engineConfig updated
         ↓
[Portfolio] → reads engineConfig → cascades to Market, Notifications, AI
         ↓
[All Modules] → react to engineConfig changes
```

## Conclusion
The Strategy Sync Engine v2 architecture extends v1 by making Portfolio the strategy control center. All modules cascade from EngineConfig via SYNC TO PORTO, ensuring user settings drive the entire application consistently. The pure function approach in `src/engine/` ensures maintainability, testability, and extensibility.
